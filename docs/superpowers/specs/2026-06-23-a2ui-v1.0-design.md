## 一、概述

在 `a2a-dev` monorepo 中实现 A2UI v1.0 协议，包括：

### 协议资源

v1.0 协议标准资源已复制到项目中的 `a2ui-protocol/specification/v1_0/`：
- `catalogs/basic/catalog.json` — 完整的基本目录定义（18 个组件 + 14 个函数 + surfaceProperties）
- `catalogs/basic/examples/*.json` — 36 个官方示例消息
- `json/common_types.json` — 通用类型定义
- `json/server_to_client.json` — 服务端→客户端信封消息 schema
- `json/client_to_server.json` — 客户端→服务端消息 schema
- `json/server_capabilities.json` — 服务端能力声明 schema
- `json/client_capabilities.json` — 客户端能力声明 schema

- A2UI 协议核心类型与 Zod v4 验证模式（`@a2a-dev/a2ui-core`）
- 框架无关渲染器核心（`@a2a-dev/a2ui-renderer`）
- A2A 扩展辅助工具（`@a2a-dev/a2ui-extension`）
- Angular v22 前端应用（`apps/a2ui-angular`）
- A2UI 示例智能体服务器（`apps/a2ui-agent`）

## 二、架构方案（方案 C — 混合优化）

参照官方实现模式，结合现有 A2A ExtensionRegistry：

- **服务端**：A2A 服务器通过 AgentCard 声明 A2UI 扩展能力，ExtensionRegistry 用于能力声明
- **智能体**：AgentExecutor 生成 A2UI 消息，打包为 Artifact 中的 DataPart（mimeType: `application/a2ui+json`）
- **传输**：使用 A2A 协议传递，通过 `A2A-Extensions` 头激活
- **客户端**：接收 A2A 响应 → 提取 DataPart → MessageProcessor 解析消息 → SurfaceModel 状态更新 → Angular 渲染

## 四、`@a2a-dev/a2ui-core` — 协议核心类型

### 4.1 消息类型（v1.0 信封）

根据 `specification/v1_0/json/server_to_client.json`，信封消息是 `oneOf` 鉴别器结构，每条消息恰好包含以下一个字段：

| 消息类型 | 描述 | v1.0 新增 |
|---------|------|----------|
| `createSurface` | 创建表面 | 支持 `components` 和 `dataModel` 内联字段（单消息 UI 实例化） |
| `updateComponents` | 更新组件 | — |
| `updateDataModel` | 更新数据模型 | — |
| `deleteSurface` | 删除表面 | — |
| `actionResponse` | 响应客户端 action | ✅ 新增 `actionId` + `value`/`error` |
| `callFunction` | 调用客户端函数 | ✅ 新增 `functionCallId` + `wantResponse` |

每条消息格式：`{ "version": "v1.0", "createSurface": {...} }`

`createSurface` 字段说明：
- `surfaceId` — 表面唯一标识符（必须全生命周期唯一）
- `catalogId` — 目录标识符
- `surfaceProperties` — 表面属性（如 `iconUrl`、`agentDisplayName`），验证于 catalog 的 `surfaceProperties` schema
- `sendDataModel` — 是否同步完整数据模型给服务器
- `components` — 可选的初始组件列表（v1.0 单消息 UI 实例化）
- `dataModel` — 可选的初始根数据模型对象（v1.0 单消息 UI 实例化）

`actionResponse` 字段说明：
- `actionId` — 对应客户端 action 的 ID
- `actionResponse.value` — 成功返回值
- `actionResponse.error` — 错误（code + message）

`callFunction` 字段说明：
- `functionCallId` — 调用实例的唯一 ID
- `callFunction` — `FunctionCall` 对象（`{ call: "...", args: {...} }`）
- `wantResponse` — 是否需要客户端返回函数结果

### 4.2 通用类型系统

根据 `specification/v1_0/json/common_types.json`，v1.0 定义了以下核心类型：

| 类型 | 说明 |
|------|------|
| `ComponentId` | 字符串，组件唯一标识符 |
| `CallId` | 字符串，函数调用唯一标识符 |
| `ComponentCommon` | 所有组件的公共属性（`id` + `accessibility`） |
| `AccessibilityAttributes` | 无障碍属性（`label`、`description`） |
| `ChildList` | 子组件列表（静态 `string[]` 或动态模板 `{ componentId, path }`） |
| `DataBinding` | 数据绑定 `{ path: string }`（JSON Pointer） |
| `DynamicValue` | 值类型：literal（string/number/boolean/array）| DataBinding | FunctionCall |
| `DynamicString` | 字符串值类型：literal string | DataBinding | FunctionCall |
| `DynamicNumber` | 数值值类型：number | DataBinding | FunctionCall |
| `DynamicBoolean` | 布尔值类型：boolean | DataBinding | FunctionCall |
| `DynamicStringList` | 字符串数组值类型：`string[]` | DataBinding | FunctionCall |
| `Action` | 交互处理器：`{ event: { name, context, wantResponse, responsePath } }` 或 `{ functionCall: FunctionCall }` |
| `CheckRule` | 校验规则 `{ condition: DynamicBoolean, message: string }` |
| `Checkable` | 支持校验的组件属性（`checks: CheckRule[]`） |
| `FunctionCall` | 函数调用 `{ call: string, args: { [key]: DynamicValue } }` |

### 4.3 基本目录组件（18 个）— 严格遵循 v1.0 规范

### 4.4 内置函数系统（14 个）

根据 `catalogs/basic/catalog.json` 的 `functions` 定义，v1.0 基本目录包含 14 个内置函数，分为四类：

#### 校验函数（用于 `CheckRule.condition`）

| 函数名 | 说明 | 参数 | 返回 |
|--------|------|------|------|
| `required` | 非 null/非空校验 | `value` | `boolean` |
| `regex` | 正则匹配校验 | `value: DynamicString`, `pattern: string` | `boolean` |
| `length` | 字符串长度校验 | `value: DynamicString`, `min?: int`, `max?: int` | `boolean` |
| `numeric` | 数值范围校验 | `value: DynamicNumber`, `min?: number`, `max?: number` | `boolean` |
| `email` | 邮箱格式校验 | `value: DynamicString` | `boolean` |

#### 格式化函数（数据转换）

| 函数名 | 说明 | 参数 | 返回 |
|--------|------|------|------|
| `formatString` | 字符串插值 `${path}` | `value: DynamicString` | `string` |
| `formatNumber` | 数字格式化 | `value: DynamicNumber`, `decimals?`, `grouping?` | `string` |
| `formatCurrency` | 货币格式化 | `value: DynamicNumber`, `currency: DynamicString`, `decimals?`, `grouping?` | `string` |
| `formatDate` | 日期格式化（Unicode TR35） | `value: DynamicValue`, `format: DynamicString` | `string` |
| `pluralize` | CLDR 复数规则 | `value: DynamicNumber`, `zero?`, `one?`, `two?`, `few?`, `many?`, `other` | `string` |

#### 操作函数

| 函数名 | 说明 | 参数 | 返回 |
|--------|------|------|------|
| `openUrl` | 打开 URL | `url: string` | `void` |

#### 逻辑函数

| 函数名 | 说明 | 参数 | 返回 |
|--------|------|------|------|
| `and` | 逻辑与 | `values: DynamicBoolean[]` (至少 2 个) | `boolean` |
| `or` | 逻辑或 | `values: DynamicBoolean[]` (至少 2 个) | `boolean` |
| `not` | 逻辑非 | `value: DynamicBoolean` | `boolean` |

#### 系统函数

| 函数名 | 说明 | 参数 | 返回 |
|--------|------|------|------|
| `@index` | 模板列表中当前项的 0 基索引 | `offset?: DynamicNumber`（默认 0） | `number` |

#### 函数调用示例

```typescript
// 在 CheckRule 中使用
{
  "checks": [{
    "condition": { "call": "regex", "args": { "value": { "path": "/email" }, "pattern": "^[\\w.]+@[\\w.]+\\.\\w+$" } },
    "message": "邮箱格式不正确"
  }]

// 在 Action 中调用客户端函数
{

    "functionCall": {
      "call": "openUrl",
      "args": { "url": "https://example.com" }

// 在 DynamicValue 中链式调用
{
  "call": "formatDate",
  "args": {
```

### 4.5 Zod v4 验证模式

- 信封验证：`A2uiMessageSchema`（z.union 鉴别器，基于 key 名）
- 组件属性：每个组件有独立 Zod schema，使用 `z.discriminatedUnion('component', [...])` 实现 `anyComponent`
- 函数：每个函数有独立 Zod schema，使用 `z.discriminatedUnion('call', [...])` 实现 `anyFunction`
- 客户端消息：`{ version: "v1.0", action: {...} }` | `{ version: "v1.0", functionResponse: {...} }` | `{ version: "v1.0", error: {...} }`
- 严格模式：使用 `.strict()` 阻止未定义属性

### 5.2 MessageProcessor 核心流程

```
processMessages(messages: A2uiMessage[])
  → for each msg:
    createSurface  → SurfaceGroupModel.addSurface(new SurfaceModel(id, catalogId))
                     + surfaceProperties / sendDataModel 设置
                     + 可选 components 和 dataModel 字段（单消息 UI 实例化）
    updateComponents → surface.componentsModel.addOrUpdate(...)
    updateDataModel  → surface.dataModel.set(path, value)
    deleteSurface    → SurfaceGroupModel.deleteSurface(id)
    actionResponse   → 通过 actionId 分发给对应的 action 发起者
    callFunction     → 查找并调用已注册的函数，如需响应则通过 functionResponse 返回
```

### 5.3 FunctionInvoker — 函数调用系统

根据 `catalog.json` 的 `functions` 定义，实现客户端函数调用引擎：

```typescript
// catalog/function-invoker.ts
export class FunctionInvoker {
  private functions = new Map<string, FunctionImpl>();

  /** 注册内置函数 */
  register(name: string, impl: FunctionImpl): void { ... }

  /** 调用函数 */
  async call(callArgs: FunctionCall): Promise<unknown> {
    const fn = this.functions.get(callArgs.call);
    if (!fn) throw new Error(`未知函数: ${callArgs.call}`);
    // 解析 args 中所有 DynamicValue 绑定
    const resolvedArgs = await this.resolveArgs(callArgs.args);
    return fn(resolvedArgs);
  }

  /** 解析 args 中的 DataBinding 和嵌套 FunctionCall */
  private async resolveArgs(args: Record<string, unknown>): Promise<Record<string, unknown>>;
}
```

内置函数注册（14 个）：
| 分类 | 函数 |
|------|------|
| 校验 | `required`, `regex`, `length`, `numeric`, `email` |
| 格式化 | `formatString`, `formatNumber`, `formatCurrency`, `formatDate`, `pluralize` |
| 操作 | `openUrl` |
| 逻辑 | `and`, `or`, `not` |
| 系统 | `@index`（特殊上下文函数，由 renderer 注入） |

### 5.4 客户端消息发送

MessageProcessor 还需负责组装客户端→服务器的消息：

### 6.3 Text 组件 Markdown 逻辑（参照官方 Angular 实现）

根据 v1.0 规范，Text 组件只有 `body`（默认）和 `caption` 两种变体。

| 变体 | Markdown 处理 | 渲染方式 |
|------|-------------|---------|
| `body` | ✅ 使用 markdown-it 渲染 | `[innerHTML]` 绑定 |
| `caption` | ❌ 纯文本 | `<em>` 标签包裹 |

核心处理逻辑：
- **异步渲染**：`effect()` 监听 text 变化 → 异步调用 markdown-it → 更新 signal
- **竞态处理**：`renderRequestId` 递增标记防竞态
- **样式**：`::ng-deep` 穿透 innerHTML（因为 markdown 生成的内容没有 Angular 作用域属性）

### 7.2 ComponentHost 动态组件渲染

核心机制：通过 Angular `ngComponentOutlet` 根据组件类型动态解析并渲染对应的 Angular 组件。

```typescript
@Component({
  selector: 'a2ui-component-host',
  template: `
    @let comp = resolvedComponent();
    @if (comp) {
      <ng-container *ngComponentOutlet="comp" />
    }
  `,
})
export class ComponentHostComponent {
  surfaceId = input.required<string>();
  componentKey = input.required<{ id: string; basePath: string }>();

  resolvedComponent = computed(() => {
    const surface = this.renderer.getSurface(this.surfaceId());
    const model = surface?.componentsModel.get(this.componentKey().id);
    if (!model) return null;
    return this.catalog.getAngularComponent(model.type);
  });
}
```

## 八、A2A 传输整合

### 8.1 数据流

```
AgentExecutor
  ├─ 生成 A2UI 消息数组 [{createSurface:...}, {updateComponents:...}]
  └─ 打包为 Artifact:
       artifact.parts[0] = {
         data: { a2uiMessages: [...] },
         mediaType: "application/a2ui+json"
       }
       ↓
A2A 传输
       ↓
A2AClient → 收到 Task → extractA2uiFromTask()
       ↓
A2uiRendererService.processMessages(a2uiMessages)
       ↓
SurfaceGroupModel 更新 → Angular 渲染
```

### 8.2 客户端→服务器 Action 数据流

用户交互产生的 action 消息通过 A2A DataPart 发送回服务器：

```
用户点击 Button
  ↓
ComponentHost → dispatchAction()
  ↓
SurfaceModel.onAction.emit(action)
  ↓
A2uiRendererService 监听 onAction
  ↓
A2AClientService 发送 DataPart 给服务器:
  message.parts[0] = {
    data: { version: "v1.0", action: {...} },
    mediaType: "application/a2ui+json"
  }
  ↓
A2A Server
  → ExtensionRegistry 在 beforeProcessMessage 中
    识别 DataPart(mimeType=a2ui+json) 的 action
  → 解析并路由到对应智能体逻辑
  → 智能体处理 action 并返回新的 A2UI 消息
```

如果客户端配置了 `sendDataModel = true`，则每次发送消息时都会在 metadata 中包含当前数据模型快照。

### 8.3 AgentCard 扩展声明

```typescript
capabilities: {
  extensions: [{
    uri: "https://a2ui.org/a2a-extension/a2ui/v1.0",
    description: "A2UI 协议 — 代理驱动的声明式 UI",
    required: false,
  }],
}
```

### 8.4 A2A 扩展规范 URI

根据 `a2ui_extension_specification.md`，此扩展的 URI 固定为：
```
https://a2ui.org/a2a-extension/a2ui/v1.0
```

激活方式：客户端在 A2A-Extensions 头中声明此 URI。

数据编码：
- A2UI 消息编码为 A2A `DataPart`
- `mediaType`（即 mimeType）：`application/a2ui+json`
- `data` 字段包含 A2UI JSON 消息的**列表**（数组格式）
- 消息列表**不是**事务单元——单条消息失败不阻止后续处理
- 渲染器应在列表中所有消息处理完毕后才重绘 UI
