# 定义自己的目录

虽然[基本目录](../../specification/v0_9/catalogs/basic/catalog.json)对入门和启动应用很有用，但大多数生产应用会定义自己的目录来反映其特定设计系统。

通过定义自己的目录，你可以限制代理只使用应用中存在的组件和视觉语言，而不是通用的输入框或按钮。

## 为什么要定义自己的目录？

每个 A2UI 表面都由一个**目录**驱动。目录只是一个 JSON Schema 文件，告诉代理哪些组件、函数和主题可供其使用。

定义自己的目录有以下好处：

- **设计系统对齐**：限制代理只使用应用中存在的组件和视觉语言。
- **安全性和类型安全**：你将整个目录注册到客户端应用中，确保只渲染受信任的组件。
- **无需映射器**：建议构建直接反映客户端设计系统的目录，而不是试图通过适配器将通用目录（如基本目录）映射到它。

基本目录只是一个示例，故意保持稀疏以便不同的渲染器易于实现。

## 工作原理

1.  **定义目录**：创建一个目录定义（JSON Schema），列出你的应用支持的组件、函数和样式。
2.  **注册目录**：将目录及其对应的组件实现（渲染器）注册到客户端应用。
3.  **宣布支持**：客户端通知代理它支持哪些目录（通过 `supportedCatalogIds`）。
4.  **代理选择目录**：代理为给定的 UI 表面选择一个目录（通过创建消息如 `createSurface` 中的 `catalogId`）。
5.  **代理生成 UI**：代理使用该目录中定义的组件名称生成组件消息。

## 实现指南

建议创建直接映射到现有组件库的目录。

=== "Web（Lit / Angular / React）"

    要在 Web 上实现自己的目录：

    - 创建一个包含组件定义的 JSON Schema。
    - 在你选择的 Web 渲染器中创建自己的 `Component` 对象和 `Catalog` 对象。
        - 向代理提供模式或引用 ID。

    _各框架的详细指南即将推出。_

=== "Flutter"

    要在 Flutter 中实现自己的目录：

    - 定义一个描述部件属性的 JSON Schema。
    - 使用自定义渲染器将模式映射到 Flutter 部件。

    _详细的 Flutter 集成指南即将推出。_

## 安全考虑

定义和注册目录时：

1.  **白名单组件**：只注册你信任的组件到目录定义中。除非严格控制，否则不要暴露具有危险能力（如执行任意脚本）的组件。
2.  **验证属性**：始终验证来自代理消息的组件属性，确保它们符合预期的类型约束。
3.  **净化文本**：除非已建立安全边界，否则避免渲染代理提供的未经净化的内容。
