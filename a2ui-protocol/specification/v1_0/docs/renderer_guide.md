# 统一架构和实现指南

本文档描述了 A2UI 客户端实现的架构。设计将关注点分离为不同层次，以最大化代码重用、确保内存安全，并在添加自定义组件时提供简化的开发者体验。

核心数据结构和渲染组件都与**目录**交互。在目录内部，实现遵循结构化的拆分：从纯粹的**组件模式**向下到绘制像素的**框架特定适配器**。

## 1. 统一架构概述

A2UI 客户端架构具有定义良好的数据流，连接了语言无关的数据结构和原生 UI 框架。

1. **A2UI 消息**从服务器到达（JSON）。
2. **`MessageProcessor`** 解析这些消息并更新 **`SurfaceModel`**（无关状态）。
3. **`Surface`**（框架入口视图）监听 `SurfaceModel` 并开始渲染。
4. `Surface` 实例化并渲染单个的 **`ComponentImplementation`** 节点以构建 UI 树。

这建立了一个根本的拆分：

- **框架无关层（数据层）**：处理 JSON 解析、状态管理、JSON 指针和模式。此逻辑在给定语言的所有 UI 框架中都是相同的。
- **框架特定层（视图层）**：处理将结构化状态转换为实际像素（React 节点、Flutter 部件、iOS 视图）。

### 实现拓扑

由于 A2UI 跨越多种语言和 UI 范式，这些架构边界的严格性和位置将根据目标生态系统而有所不同。

#### 动态语言（例如 TypeScript / JavaScript）

在高度动态的生态系统中，架构通常跨多个包拆分以最大化不同 UI 框架（React、Angular、Vue、Lit）之间的代码重用。

#### 静态语言（例如 Kotlin、Swift、Dart）

在静态类型语言中，运行时反射通常受到限制或出于性能原因而不被鼓励。

#### 组合核心 + 框架库（例如 Swift + SwiftUI）

在由单一 UI 框架主导的生态系统中，开发者通常构建统一的库而不是将核心和框架拆分为单独的包。

## 2. 核心接口

在 A2UI 架构的核心，有五个关键接口将数据连接到屏幕。

### `ComponentApi`

组件的框架无关定义。它定义了组件的名称和精确的 JSON 模式足迹，没有任何渲染逻辑。

### `ComponentImplementation`

渲染组件的框架特定逻辑。它扩展了 `ComponentApi`，包含 `build` 或 `render` 方法。

### `Surface`

特定框架的入口点部件/视图。它用 `SurfaceModel` 实例化。

### `SurfaceModel` & `ComponentContext`

状态容器。`SurfaceModel` 表示单个 UI 表面的整个状态，持有 `DataModel` 和扁平的组件配置列表。`ComponentContext` 是由 `Surface` 创建并在渲染期间传递给 `ComponentImplementation` 的临时对象。

## 3. 核心数据层（详细规范）

数据层维护一个长期存在的、可变的状态对象。此层在所有编程语言中遵循完全相同的设计，并且在移植到新框架时**不需要设计工作**。
