# A2A 中的智能体发现

要使用 Agent2Agent（A2A）协议进行协作，AI 智能体需要首先找到彼此并了解彼此的能力。A2A 通过 **[智能体卡片](../specification.md#5-agent-discovery-the-agent-card)** 标准化了智能体的自我描述。然而，这些智能体卡片的发现方法因环境和要求而异。智能体卡片定义了智能体提供的内容。存在多种策略供客户端智能体发现这些卡片。策略的选择取决于部署环境和安全要求。

## 智能体卡片的作用

智能体卡片是一个 JSON 文档，作为 A2A 服务器（远程智能体）的数字"名片"。它对智能体发现和交互至关重要。智能体卡片中包含的关键信息如下：

- **身份：** 包括 `name`、`description` 和 `provider` 信息。
- **服务端点：** 指定 A2A 服务的 `url`。
- **A2A 能力：** 列出支持的功能，如 `streaming` 或 `pushNotifications`。
- **身份验证：** 详细说明所需的 `schemes`（例如，"Bearer"、"OAuth2"）。
- **技能：** 使用 `AgentSkill` 对象描述智能体的任务，包括 `id`、`name`、`description`、`inputModes`、`outputModes` 和 `examples`。

客户端智能体使用智能体卡片来确定智能体的适用性、构建请求并确保安全通信。

## 发现策略

以下各节详细说明了客户端智能体用于发现远程智能体卡片的常见策略：

### 1. Well-Known URI

这种方法推荐用于公共智能体或意图在特定领域内广泛发现的智能体。

- **机制：** A2A 服务器通过在其域的标准 `well-known` URI 上托管其智能体卡片使其可发现。标准路径是 `https://{agent-server-domain}/.well-known/agent-card.json`，遵循 [RFC 8615](https://datatracker.ietf.org/doc/html/rfc8615) 的原则。

- **过程：**
    1. 客户端智能体知道或通过编程发现潜在 A2A 服务器的域（例如，`smart-thermostat.example.com`）。
    2. 客户端执行 HTTP GET 请求到 `https://smart-thermostat.example.com/.well-known/agent-card.json`。
    3. 如果智能体卡片存在且可访问，服务器将其作为 JSON 响应返回。

- **优点：**
    - 易于实现
    - 符合标准
    - 促进自动发现

- **注意事项：**
    - 最适合开放或域控制的发现场景。
    - 如果智能体卡片包含敏感详细信息，则需要在提供智能体卡片的端点进行身份验证。

### 2. 精选注册中心（基于目录的发现）

这种方法用于企业环境或公共市场，其中智能体卡片通常由中央注册中心管理。精选注册中心充当中央仓库，允许客户端基于"技能"或"标签"等条件查询和发现智能体。

- **机制：** 一个中介服务（注册中心）维护智能体卡片的集合。客户端查询此注册中心以根据各种条件（例如，提供的技能、标签、提供商名称、能力）查找智能体。

- **过程：**
    1. A2A 服务器将其智能体卡片发布到注册中心。
    2. 客户端智能体查询注册中心的 API，并按"特定技能"等条件进行搜索。
    3. 注册中心返回匹配的智能体卡片或引用。

- **优点：**
    - 集中管理和治理。
    - 基于能力的发现（例如，按技能）。
    - 支持访问控制和信任框架。
    - 适用于私有和公共市场。
- **注意事项：**
    - 需要部署和维护注册中心服务。
    - 当前的 A2A 规范没有为精选注册中心规定标准 API。

### 3. 直接配置/私有发现

这种方法用于紧密耦合的系统、私有智能体或开发目的，其中客户端直接配置有智能体卡片信息或 URL。

- **机制：** 客户端应用程序使用硬编码的详细信息、配置文件、环境变量或专有 API 进行发现。
- **过程：** 过程特定于应用程序的部署和配置策略。
- **优点：** 这种方法对于在已知的、静态的关系中建立连接是直接的。
- **注意事项：**
    - 对于动态发现场景不灵活。
    - 智能体卡片信息的更改需要客户端重新配置。
    - 基于专有 API 的发现也缺少标准化。

## 保护智能体卡片

智能体卡片包含敏感信息，例如：

- 内部或受限智能体的 URL。
- 敏感技能的描述。

### 保护机制

为减轻风险，应考虑以下保护机制：

- **已验证的智能体卡片：** 我们建议对敏感信息使用[已验证的扩展智能体卡片](../specification.md#3111-get-extended-agent-card)，或用于提供更详细版本的卡片。
- **安全端点：** 在提供智能体卡片的 HTTP 端点（例如，`/.well-known/agent-card.json` 或注册中心 API）上实施访问控制。方法包括：
    - 双向 TLS（mTLS）
    - 网络限制（例如，IP 范围）
    - HTTP 身份验证（例如，OAuth 2.0）

- **注册中心选择性披露：** 注册中心根据客户端的身份和权限返回不同的智能体卡片。

任何包含敏感数据的智能体卡片都必须通过身份验证和授权机制进行保护。A2A 规范强烈建议使用带外动态凭证，而不是在智能体卡片中嵌入静态密钥。

## 缓存注意事项

智能体卡片描述了智能体的能力，通常变化不频繁——例如，当添加技能或更新身份验证要求时。对智能体卡片端点应用标准 HTTP 缓存实践可以减少不必要的网络请求，同时确保客户端最终收到更新后的信息。

### 服务器指南

托管智能体卡片端点的服务器应在其响应中包含 HTTP 缓存头。带有适当 `max-age` 指令的 `Cache-Control` 头允许客户端和中介缓存卡片一段指定的时间。包含 `ETag` 头（从卡片的 `version` 字段或内容哈希派生）使客户端能够进行条件请求，避免重新下载未更改的卡片。

### 客户端指南

获取智能体卡片的客户端应遵循标准 HTTP 缓存语义。当缓存的卡片过期时，客户端应使用条件请求（例如，带有存储的 `ETag` 的 `If-None-Match` 或 `If-Modified-Since`），而不是无条件地重新获取完整卡片。当服务器未提供缓存头时，客户端可以应用合理的默认缓存持续时间。

对于扩展智能体卡片，客户端还应遵循[规范](../specification.md#133-extended-agent-card-access-control)中描述的会话范围缓存指南。

有关规范性要求，请参见规范的[第 8.6 节](../specification.md#86-caching)。
