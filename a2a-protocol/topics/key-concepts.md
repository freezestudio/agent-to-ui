# A2A 的核心概念和组件

A2A 使用一组核心概念来定义智能体如何交互。
理解这些核心构建块有助于开发或集成符合 A2A 的系统。

![A2A 参与者示意图：用户、A2A 客户端（客户端智能体）和 A2A 服务器（远程智能体）](../assets/a2a-actors.png){ width="70%" style="margin:20px auto;display:block;" }

## A2A 交互中的核心参与者

- **用户**：最终用户，可以是人类操作员或自动化服务。用户发起请求或定义需要从一个或多个 AI 智能体获得帮助的目标。
- **A2A 客户端（客户端智能体）**：代表用户行事的应用程序、服务或另一个 AI 智能体。客户端使用 A2A 协议发起通信。
- **A2A 服务器（远程智能体）**：暴露实现 A2A 协议的 HTTP 端点的 AI 智能体或智能体系统。它接收来自客户端的请求，处理任务，并返回结果或状态更新。从客户端的角度来看，远程智能体作为一个_不透明的_（黑盒）系统运行，意味着其内部工作机制、记忆或工具不会暴露。

## 基本通信元素

下表描述了 A2A 中的基本通信元素：

| 元素 | 描述 | 关键目的 |
| :------ | :---------- | :---------- |
| 智能体卡片（Agent Card） | 描述智能体身份、能力、端点、技能和身份验证要求的 JSON 元数据文档。 | 使客户端能够发现智能体并理解如何安全有效地与之交互。 |
| 任务（Task） | 由智能体发起的有关状态的工作单元，具有唯一 ID 和已定义的生命周期。 | 促进长时间运行操作的跟踪，实现多轮交互和协作。 |
| 消息（Message） | 客户端与智能体之间的单轮通信，包含内容和角色（"user"或"agent"）。 | 传递指令、上下文、问题、答案或状态更新，这些不一定是正式的制品。 |
| 部件（Part） | 消息和制品中使用的基本内容容器。一个部件包含以下之一：文本内容、文件引用（URL 或内联字节）或结构化数据。 | 为智能体提供灵活性，使其能够在消息和制品中交换各种内容类型。 |
| 制品（Artifact） | 智能体在任务期间生成的有形输出（例如，文档、图像或结构化数据）。 | 传递智能体工作的具体结果，确保结构化和可检索的输出。 |

## 交互机制

A2A 协议支持各种交互模式，以适应不同的响应性和持久性需求。这些机制确保智能体能够高效可靠地交换信息，无论任务的复杂性或持续时间如何：

- **请求/响应（轮询）**：客户端发送请求，服务器响应。对于长时间运行的任务，客户端定期轮询服务器以获取更新。
- **使用服务器发送事件（SSE）的流式传输**：客户端发起流，通过开放的 HTTP 连接实时接收来自服务器的增量结果或状态更新。
- **推送通知**：对于非常长时间运行的任务或断连场景，服务器可以在任务状态发生重大更新时主动向客户端提供的 Webhook 发送异步通知。

有关流式传输和推送通知的详细探索，请参考[流式传输与异步操作](./streaming-and-async.md)文档。

## 智能体卡片（Agent Card）

智能体卡片是一个 JSON 文档，作为数字名片用于初始发现和交互设置。它提供关于智能体的基本元数据。客户端解析这些信息以确定智能体是否适合给定的任务、如何构建请求以及如何安全地通信。关键信息包括身份、服务端点（URL）、A2A 能力、身份验证要求和技能列表。

## 消息和部件

消息代表客户端与智能体之间的单轮通信。它包括一个角色（"user"或"agent"）和一个唯一的 `messageId`。它包含一个或多个部件（Part）对象，这些是实际内容的细粒度容器。这种设计使 A2A 能够做到模态无关。

`Part` 对象是一个灵活的容器，可以使用 `oneof` 字段结构保存不同类型的内容。一个部件必须恰好包含以下内容字段之一：

- `text`：包含纯文本内容的字符串。
- `raw`：包含二进制文件数据的字节数组（内联）。
- `url`：引用外部文件内容的字符串 URI。
- `data`：用于机器可读数据的结构化 JSON 值（例如，对象、数组）。

此外，每个 `Part` 可以包含：

- `mediaType`：内容的 MIME 类型（例如，`"text/plain"`、`"image/png"`、`"application/json"`）。
- `filename`：文件或内容的可选名称。
- `metadata`：用于附加上下文的键值映射。

## 制品（Artifact）

制品代表远程智能体在任务处理期间生成的有形输出或具体结果。与普通消息不同，制品是实际的可交付成果。制品具有唯一的 `artifactId`、人类可读的名称，并由一个或多个部件对象组成。制品与任务生命周期紧密相关，可以增量方式流式传输到客户端。

## 智能体响应：任务或消息

智能体响应可以是一个新的`任务`（Task）（当智能体需要执行长时间运行的操作时）或一条`消息`（Message）（当智能体可以立即响应时）。

有关详细信息，请参见[任务的生命周期](./life-of-a-task.md)。

## 其他重要概念

- **上下文（`contextId`）：** 服务器生成的标识符，可用于逻辑上将多个相关的`任务`（Task）对象分组，在一系列交互中提供上下文。
- **传输和格式：** A2A 通信通过 HTTP(S) 进行。JSON-RPC 2.0 用作所有请求和响应的负载格式。
- **身份验证与授权：** A2A 依赖于标准的 Web 安全实践。身份验证要求在智能体卡片中声明，凭据（例如，OAuth 令牌、API 密钥）通常通过 HTTP 头传递，与 A2A 协议消息本身分开。有关更多信息，请参见[企业级特性](./enterprise-ready.md)。
- **智能体发现：** 客户端查找智能体卡片以了解可用 A2A 服务器及其能力的过程。有关更多信息，请参见[智能体发现](./agent-discovery.md)。
- **扩展：** A2A 允许智能体在其 AgentCard 中声明自定义协议扩展。有关更多信息，请参见[扩展](./extensions.md)。
