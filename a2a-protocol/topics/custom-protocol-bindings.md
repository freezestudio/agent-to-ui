# 自定义协议绑定

A2A 协议附带三个标准绑定（JSON-RPC、gRPC 和 HTTP+JSON/REST），覆盖了大多数部署场景。自定义协议绑定允许实现者通过标准集合未涵盖的额外传输机制暴露 A2A 操作。

自定义协议绑定是一个互补但不同的概念，与[扩展](extensions.md)不同。扩展通过添加新数据、方法或状态转换来修改协议交互的*行为*。自定义协议绑定更改*传输层*本身——例如，通过 WebSocket 暴露 A2A 以实现低延迟双向通信，或通过 MQTT 为连接受限的 IoT 环境暴露 A2A。

## 在智能体卡片中声明

自定义协议绑定在智能体卡片的 `supportedInterfaces` 列表中声明。每个条目通过 URI 标识传输、端点 URL 和它实现的 A2A 协议版本。`protocolBinding` 字段应是一个唯一标识该绑定的 URI（有关规范性要求和版本化指导，请参见[规范的第 5.8 节](../specification.md#58-custom-binding-identification)）。

```json
{
  "supportedInterfaces": [
    {
      "url": "wss://agent.example.com/a2a/websocket",
      "protocolBinding": "https://a2a-protocol.org/bindings/websocket",
      "protocolVersion": "1.0"
    }
  ]
}
```

支持多个绑定的智能体列出所有绑定。客户端按顺序解析 `supportedInterfaces` 并选择它们支持的第一个传输，因此条目应按偏好顺序列出。

## 要求

自定义协议绑定必须符合规范中[协议绑定要求和互操作性](../specification.md#5-protocol-binding-requirements-and-interoperability)部分的所有要求。特别是：

- **必须支持所有核心操作。** 绑定必须暴露抽象操作层中定义的每个操作（发送消息、获取任务、取消任务、流式传输、推送通知等）。
- **必须保留数据模型。** 所有数据结构必须在功能上等同于规范的 Protocol Buffer 定义。JSON 序列化必须使用 camelCase 字段名，时间戳必须是 UTC 的 ISO 8601 字符串。
- **行为必须一致。** 语义等效的请求必须产生语义等效的结果，无论使用哪个绑定。

## 需要指定的关键领域

自定义绑定规范必须解决以下每个领域。

### 数据类型映射

记录每种 Protocol Buffer 类型在自定义传输中的表示方式，包括：

- 二进制数据编码（例如，基于文本的传输使用 base64）
- 枚举表示（字符串、整数或命名常量）
- 时间戳格式（按照核心约定使用 UTC 的 ISO 8601 字符串）

### 服务参数

服务参数是用于携带横向适用上下文（如跟踪标识符或身份验证提示）的键值对。绑定规范必须说明：

- 用于携带服务参数的机制（例如，自定义消息头、顶级元数据字段）
- 对键和值的任何字符编码或大小限制
- 绑定本身保留的任何名称

对于缺乏原生头支持的传输，常见模式是将服务参数作为 JSON 对象嵌入到专用的元数据字段中，例如 `a2a-service-parameters`。

### 错误映射

绑定必须将所有 A2A 错误类型映射到传输原生的错误表示，同时保留其语义含义。提供一个映射表，等同于规范中[错误代码映射](../specification.md#54-error-code-mappings)部分的表格，显示每种 A2A 错误类型（例如，`TaskNotFoundError`、`UnsupportedOperationError`）如何在自定义绑定的原生错误格式中表达。

### 流式传输

如果传输支持流式传输，请记录：

- 流机制（例如，WebSocket 帧、分块编码、长轮询）
- 排序保证（事件必须按生成顺序传递）
- 连接中断时的重新连接行为
- 流完成或终止如何向客户端发出信号

如果传输不支持流式传输，请在智能体卡片中明确说明此限制，以便客户端可以回退到轮询。

### 身份验证和授权

记录智能体卡片中声明的身份验证凭证如何使用自定义传输传输。定义身份验证挑战如何传达给客户端，并确保自定义绑定不会无意中绕过智能体的主要安全控制。

## 互操作性测试

在发布自定义绑定之前，请验证：

- 所有操作对于相同的逻辑请求行为与标准绑定相同
- 错误条件、大负载和长时间运行的任务处理正确
- 与标准绑定行为的任何有意偏差都已清晰记录
- 规范中包含示例请求和响应以帮助实现者

## 治理

A2A 组织使用正式的治理框架来规定自定义协议绑定的提议、开发、推进和维护方式。官方绑定使用 `https://a2a-protocol.org/bindings/` URI 前缀，并托管在 `a2aproject` 组织下，使用 `cpb-` 仓库前缀（实验性绑定使用 `experimental-cpb-`）。A2A SDK **应该**实现官方的自定义协议绑定。

有关完整的治理流程——包括层级、生命周期、SDK 支持和法律要求——请参见[扩展和协议绑定治理](extension-and-binding-governance.md)页面。
