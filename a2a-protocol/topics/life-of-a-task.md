# 任务的生命周期

在 Agent2Agent（A2A）协议中，交互范围从简单的、无状态的交换到复杂的、长时间运行的流程。当智能体收到来自客户端的消息时，它可以通过两种基本方式之一进行响应：

- **以无状态的`消息`（Message）响应**：这种响应类型通常用于即时的、自包含的交互，完成后无需进一步的状态管理。
- **启动有状态的`任务`（Task）**：如果响应是一个`任务`，智能体将按照已定义的生命周期处理它，根据需要沟通进度和要求输入，直到达到中断状态（例如，`input-required`、`auth-required`）或终止状态（例如，`completed`、`canceled`、`rejected`、`failed`）。

## 分组相关交互

`contextId` 是一个关键标识符，它在逻辑上对多个`任务`（Task）对象和独立的`消息`（Message）对象进行分组，在一系列交互中提供连续性。

- 当客户端首次发送消息时，智能体响应一个新的 `contextId`。如果启动了任务，它还将有一个 `taskId`。
- 客户端可以发送后续消息并包含相同的 `contextId`，以表示它们在同一上下文中继续先前的交互。
- 客户端可选择将 `taskId` 附加到后续消息中，以表示该消息继续该特定任务。

`contextId` 使得跨多个可能并发的任务朝着共同目标或共享上下文会话进行协作成为可能。在内部，A2A 智能体（尤其是使用 LLM 的智能体）使用 `contextId` 来管理其内部对话状态或 LLM 上下文。

## 智能体响应：消息或任务

选择以`消息`还是`任务`响应取决于交互的性质和智能体的能力：

- **琐碎交互使用消息**：`消息`对象适用于不需要长时间运行处理或复杂状态管理的事务性交互。智能体可以使用消息来协商任务的接受或范围，然后再提交到`任务`对象。
- **有状态交互使用任务**：一旦智能体将来消息的意图映射到需要较长时间内进行大量、可跟踪工作的支持能力，智能体就以`任务`对象响应。

从概念上讲，智能体以不同的复杂级别运行：

- **纯消息智能体**：始终以`消息`对象响应。它们通常不管理复杂状态或长时间运行的执行，并使用 `contextId` 将消息联系在一起。这些智能体可能直接包装 LLM 调用和简单工具。
- **任务生成智能体**：始终以`任务`对象响应，即使对于响应也是如此，这些响应随后被建模为已完成的任务。一旦创建了任务，智能体将只返回`任务`对象作为发送消息的响应，一旦任务完成，就不能再发送消息。这种方法避免了在`任务`与`消息`之间做决定，但即使对于简单的交互也会创建已完成的任务对象。
- **混合智能体**：同时生成`消息`和`任务`对象。这些智能体使用消息来协商智能体能力和任务的工作范围，然后发送`任务`对象来跟踪执行并管理诸如此类的状态 `input-required` 或错误处理。一旦创建了任务，智能体将只返回`任务`对象作为发送消息的响应，一旦任务完成，就不能再发送消息。混合智能体使用消息来协商任务的范围，然后生成任务来跟踪其执行。
    有关混合智能体的更多信息，请参见 [A2A 协议：解构任务与消息](https://discuss.google.dev/t/a2a-protocol-demystifying-tasks-vs-messages/255879)。

## 任务细化

客户端通常需要基于任务结果发送新请求或细化先前任务的输出。这通过使用与原始任务相同的 `contextId` 启动另一个交互来建模。客户端通过提供对原始任务的引用（在`消息`对象中使用 `referenceTaskIds`）进一步提示智能体。然后智能体以新`任务`或`消息`响应。

## 任务不可变性

一旦任务达到终止状态（completed、canceled、rejected 或 failed），它就无法重新启动。任何与该任务相关的后续交互（如细化）必须在相同的 `contextId` 内启动新任务。这一原则提供了几个好处：

- **任务不可变性。** 客户端可以可靠地引用任务及其关联的状态、制品和消息，提供输入到输出的清晰映射。这对于编排和可追溯性很有价值。
- **清晰的工作单元。** 每个新请求、细化或后续操作都成为一个独特的任务。这简化了记账，允许对智能体工作进行细粒度跟踪，并使每个制品可追溯到特定的工作单元。
- **更易实现。** 这消除了智能体开发者在创建新任务还是重新启动现有任务之间的歧义。

## 并行后续操作

A2A 通过使智能体能够为在同一 `contextId` 内发送的每个后续消息创建不同的并行任务来支持并行工作。这使得客户端能够跟踪各个任务，并在先决任务完成后立即创建新的依赖任务。

例如：

- 任务 1：预订飞往赫尔辛基的航班。
- 任务 2：基于任务 1，预订酒店。
- 任务 3：基于任务 1，预订雪地摩托活动。
- 任务 4：基于任务 2，为酒店预订添加水疗预约。

## 引用先前的制品

服务智能体从引用的任务或 `contextId` 推断相关制品。作为领域专家，服务智能体最擅长解决歧义或识别缺失信息。如果存在歧义，智能体通过返回 `input-required` 状态向客户端请求澄清。然后客户端在其响应中指定制品，可选择在 `Part` 元数据中填充制品引用（`artifactId`、`taskId`）。

## 跟踪制品变更

后续或细化任务通常会导致基于旧制品创建新制品。跟踪这些变更很重要，以确保在后续交互中只使用制品的最新版本。这可以概念化为版本历史，其中每个新制品都链接到其前驱。

然而，客户端处于管理此制品链接的最佳位置。客户端决定什么是可接受的结果，并有权接受或拒绝新版本。因此，服务智能体不应负责跟踪制品变更，此链接不是 A2A 协议规范的一部分。客户端应在其端维护此版本历史，并向用户呈现最新的可接受版本。

为便于客户端跟踪，服务智能体在生成现有制品的细化版本时应使用一致的 `artifact-name`。

在启动后续或细化任务时，客户端应明确引用他们打算细化的特定制品，最好是来自其视角的"最新"版本。如果未提供制品引用，服务智能体可以：

- 尝试基于当前 `contextId` 推断预期的制品。
- 如果存在歧义或上下文不足，智能体应以 `input-required` 任务状态响应，以向客户端请求澄清。

## 后续操作示例场景

以下示例说明了典型的带有后续操作的任务流程：

1. 客户端向智能体发送消息：

    ```json
    {
      "jsonrpc": "2.0",
      "id": "req-001",
      "method": "SendMessage",
      "params": {
        "message": {
          "role": "user",
          "parts": [
            {
              "text": "生成一张海洋上的帆船图像。"
            }
          ],
          "messageId": "msg-user-001"
        }
      }
    }
    ```

2. 智能体以帆船图像响应（任务已完成）：

    ```json
    {
      "jsonrpc": "2.0",
      "id": "req-001",
      "result": {
        "task": {
          "id": "task-boat-gen-123",
          "contextId": "ctx-conversation-abc",
          "status": {
            "state": "TASK_STATE_COMPLETED"
          },
          "artifacts": [
            {
              "artifactId": "artifact-boat-v1-xyz",
              "name": "sailboat_image.png",
              "description": "一张海洋上帆船的生成图像。",
              "parts": [
                {
                  "filename": "sailboat_image.png",
                  "mediaType": "image/png",
                  "raw": "base64_encoded_png_data_of_a_sailboat"
                }
              ]
            }
          ]
        }
      }
    }
    ```

3. 客户端要求将船改为红色。此细化请求引用先前的 `taskId` 并使用相同的 `contextId`。

    ```json
    {
      "jsonrpc": "2.0",
      "id": "req-002",
      "method": "SendMessage",
      "params": {
        "message": {
          "role": "user",
          "messageId": "msg-user-002",
          "contextId": "ctx-conversation-abc",
          "referenceTaskIds": [
            "task-boat-gen-123"
          ],
          "parts": [
            {
              "text": "请将帆船修改为红色。"
            }
          ]
        }
      }
    }
    ```

4. 智能体以新的图像制品响应（新任务，相同上下文，相同制品名称）：智能体在相同的 `contextId` 内创建新任务。新的帆船图像制品保留相同的名称但具有新的 `artifactId`。

    ```json
    {
      "jsonrpc": "2.0",
      "id": "req-002",
      "result": {
        "task": {
          "id": "task-boat-color-456",
          "contextId": "ctx-conversation-abc",
          "status": {
            "state": "TASK_STATE_COMPLETED"
          },
          "artifacts": [
            {
              "artifactId": "artifact-boat-v2-red-pqr",
              "name": "sailboat_image.png",
              "description": "一张海洋上红色帆船的生成图像。",
              "parts": [
                {
                  "filename": "sailboat_image.png",
                  "mediaType": "image/png",
                  "raw": "base64_encoded_png_data_of_a_RED_sailboat"
                }
              ]
            }
          ]
        }
      }
    }
    ```
