# 扩展和协议绑定治理

A2A 组织对[扩展](extensions.md)和[自定义协议绑定](custom-protocol-bindings.md)使用统一的治理框架。本文档定义了在 A2A 组织内提议、开发、推进和维护这些制品的正式流程。

任何人都可以独立开发和发布扩展或自定义协议绑定。本文描述的层级和生命周期特别适用于托管在 `a2aproject` GitHub 组织下的那些。

## 层级

扩展和自定义协议绑定在 `a2aproject` 组织内都使用两级系统。仓库命名和 URI 前缀因类型而异：

|                          | 扩展                                          | 自定义协议绑定                             |
| :----------------------- | :---------------------------------------------- | :--------------------------------------------- |
| 官方仓库前缀             | `ext-{名称}`                                    | `cpb-{名称}`                                   |
| 实验性仓库前缀           | `experimental-ext-{名称}`                       | `experimental-cpb-{名称}`                      |
| 官方 URI 前缀            | `https://a2a-protocol.org/extensions/`          | `https://a2a-protocol.org/bindings/`           |

### 官方

官方制品在 `a2aproject` GitHub 组织下开发和维护，由 TSC 官方推荐。每个仓库都有在 `MAINTAINERS.md` 中指定的维护者。

**要求：**

- 规范必须使用与核心规范相同的语言（[RFC 2119](https://tools.ietf.org/html/rfc2119)）
- 必须在 Apache 2.0 许可下授权
- 必须至少有一个参考实现
- 应该在 A2A 网站上有相关文档

### 实验性

实验性制品为社区贡献者提供了孵化路径，以便在升级到官方状态之前进行原型设计和协作。

**创建要求：**

- 实验性仓库只能在 A2A 维护者的赞助下创建
- 赞助的维护者负责实验性制品的初始监督
- 实验性仓库必须在 README 中明确说明其实验性/非官方状态
- 任何发布的包必须使用明确指示实验性状态的命名
- TSC 保留监督权，包括归档或删除实验性仓库的能力

## 生命周期

扩展和自定义协议绑定经历以下阶段。

### 提议阶段

任何社区成员都可以提议扩展或自定义协议绑定：

1. **打开问题**：在主要的 `a2aproject/A2A` 仓库中创建一个问题，描述：
    - 描述扩展目的或绑定的传输和用例的摘要
    - 解释为什么这不能通过核心协议或现有标准绑定实现的动机
    - 初始技术方法或规范草案
2. **社区讨论**：提议开放供社区反馈和完善

### 维护者赞助

为了使提议进入实验性状态：

1. **获得赞助者**：A2A 维护者必须同意赞助该提议
2. **创建仓库**：赞助的维护者在 `a2aproject` 下创建 `experimental-ext-*` 或 `experimental-cpb-*` 仓库
3. **监督**：赞助的维护者提供初始监督并确保与 A2A 设计原则一致

### 实验性开发

在实验性状态期间：

- 贡献者迭代规范和完善参考实现
- 实验性制品可以由早期采用者使用，但理解会发生重大变更
- 收集和整合社区反馈
- 实验性仓库必须清楚指示其非官方状态

### 升级到官方状态

要将实验性扩展或绑定升级到官方状态：

1. **成熟度要求**：
    - 至少一个生产质量的参考实现
    - 符合 A2A 标准的文档
    - 社区采用或兴趣的证据
    - 对持续维护的明确维护者承诺
2. **升级提议**：在 `a2aproject/A2A` 中打开一个问题，包含：
    - 对实验性仓库及其实现的引用
    - 社区反馈和采用的总结
    - 官方制品的建议维护者
3. **TSC 投票**：
    - 提议被添加到 TSC 会议议程
    - **法定人数要求**：至少 50% 的 TSC 投票成员必须出席
    - **批准**：需要出席会议的多数票（根据 A2A 治理）
    - TSC 可能在最终投票前要求修订
4. **接受**：
    - （扩展）仓库从 `experimental-ext-*` 重命名为 `ext-*`；文档添加到 A2A 网站的扩展页面
    - （自定义协议绑定）仓库从 `experimental-cpb-*` 重命名为 `cpb-*`；文档添加到 A2A 网站的自定义协议绑定页面

### 官方迭代

一旦成为官方，扩展和绑定可以进行迭代：

- 仓库维护者负责日常治理
- 更改应通过相关工作组协调（如果存在）
- 重大变更需要新的标识符
- 重大变更需要 TSC 审查
- 维护者应与 SDK 维护者协调实现更新

### 升级到核心协议

某些扩展可能最终过渡到核心协议功能，某些自定义协议绑定可能过渡到核心绑定。这通过现有的 A2A 规范增强流程进行治理：

- 按照标准规范变更流程提交提议
- 提议引用官方扩展或绑定及其采用情况
- 适用 TSC 投票，需满足标准法定人数和多数要求
- 并非所有扩展或绑定都适合包含在核心中；许多将无限期地作为扩展或自定义绑定保留

## SDK 支持

扩展和官方自定义协议绑定的 SDK 支持要求不同，反映了它们在协议生态系统中的不同角色。

**扩展**：A2A SDK 可以实现扩展。如果实现：

- 扩展必须在默认情况下禁用，并需要明确选择加入
- SDK 文档应列出支持的扩展
- SDK 维护者对扩展支持决策拥有完全自主权
- 协议一致性不需要扩展支持

**官方自定义协议绑定**：A2A SDK 应该实现官方自定义协议绑定。如果实现：

- 自定义协议绑定必须在默认情况下禁用，并需要明确选择加入
- SDK 文档应列出支持的自定义协议绑定
- SDK 维护者对绑定支持决策拥有完全自主权
- 协议一致性不需要自定义协议绑定支持

## 法律要求

### 许可

官方扩展和自定义协议绑定必须在 Apache 2.0 许可下可用，与核心 A2A 项目一致。

### 贡献者许可授予

通过向官方 A2A 扩展或自定义协议绑定仓库提交贡献，贡献者表示：

1. 他们拥有授予这些权利的法律权力
2. 该贡献是原创作品，或者他们有足够的权利提交它
3. 他们授予 Linux 基金会和接收者永久的、全球性的、非排他性的、免版税的许可，以使用、复制、修改和分发该贡献

### 反垄断

扩展和自定义协议绑定开发者承认：

- 他们可能与其他参与者竞争
- 他们没有实施任何扩展或绑定的义务
- 他们可以自由开发竞争性扩展或绑定
- 作为官方扩展或绑定的状态不创建排他性关系
