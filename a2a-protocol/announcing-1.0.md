# A2A 协议发布 v1.0：智能体间通信的生产就绪标准

A2A 协议社区今天宣布发布 A2A 协议 v1.0，标志着 AI 智能体之间通信开放标准的第一个稳定、生产就绪版本。该协议由来自八家主要科技公司的代表组成的技术指导委员会指导。

随着组织构建越来越复杂的多智能体系统，互操作性已成为决定性挑战。团队可以在单个平台内有效协调智能体，但跨技术栈和组织边界连接这些系统仍然困难。A2A 通过结合对多种协议绑定的支持、无缝版本协商和通用语义模型来解决这一挑战，使智能体能够以可预测的行为跨系统互操作。

v1.0 版本强调成熟而非重新发明：核心思想保持不变，同时去除了粗糙的边缘，澄清了模糊的领域，并更直接地解决了企业部署需求。官方 SDK 确保 v1.0 A2A 智能体与旧版本无缝协作。

## 实现企业需求

v1.0 版本引入了若干针对生产环境的能力，在这些环境中，信任、规模和运营控制是不可妥协的。

- **异构环境支持**通过多协议绑定和版本协商实现跨不同技术栈的互操作性，使企业不受限于单一供应商或平台。
- **多租户支持**允许单个端点安全地托管多个智能体。
- **签名的智能体卡片**提供智能体身份和元数据的加密验证，在跨组织边界交互之前建立信任。
- **改进的安全态势**现代化了安全流程，移除了不再符合当前最佳实践的遗留模式。

这些变化共同将 A2A 从早期采用者实施推向更广泛的企业信心，特别是在受监管或多方场景中。

## 面向 Web 的架构以支持规模

A2A v1.0 与 Web 的核心架构原则保持一致：无状态、分层架构、标准协议绑定和基础设施友好的通信模式。这种一致性在运营上很重要，因为组织可以使用他们已经用于 Web 系统的经过验证的负载均衡、网关、安全和可观测性模式来扩展智能体交互。

该标准建立在行业验证的协议之上，包括 JSON+HTTP、gRPC 和 JSON-RPC。它还保持了较低的入门门槛：在最简单的形式中，A2A 交互可以从单个 HTTP 请求开始。

A2A 还为消费者提供了接收结果的灵活性。根据工作负载和运营需求，客户端可以使用轮询、流式传输或 Webhook 来消费任务更新和响应。

## 与 MCP 互补，而非替代

该版本还强化了 A2A 与模型上下文协议（MCP）的关系，这一点在早期的生态系统讨论中引起了混淆。

MCP 和 A2A 解决不同层次的问题。MCP 通常用于单个智能体级别的工具和上下文集成。A2A 专注于智能体之间的通信和协调。在实践中，许多系统将同时使用两者：MCP 在智能体内部，A2A 在智能体之间。

## 从早期版本平滑迁移

v1.0 版本收紧了规范行为，其中包括交互协议中的重大变更。然而，AgentCard 以向后兼容的方式演变，现在允许智能体同时宣传对现有 v0.3 协议行为和 v1.0 的支持。这使得客户端可以逐步迁移，而不是通过单一切换。

这种方法旨在保护现有投资，同时仍然提供更清洁、更持久的标准的好处。

## 为什么这个版本现在很重要

AI 智能体越来越多地部署在部门、产品和合作伙伴生态系统中。在这个规模上，关键问题不再是智能体是否可以在一个栈内协调，而是它们是否可以可靠地跨组织和平台边界协作。开放协议决定了组织是能够组合最佳系统还是被锁定在孤立的栈中。

A2A v1.0 为市场提供了开放多智能体协作的更坚实基础。社区现在专注于提供多语言 v1.0 SDK 支持，以帮助开发者轻松构建合规解决方案。

完整的 v1.0 材料，包括规范和迁移文档，可在 [a2a-protocol.org](https://a2a-protocol.org) 和 [GitHub](https://github.com/a2aproject/A2A) 获取。

**关于 A2A 协议**
Agent-to-Agent（A2A）协议是一种开放标准，使 AI 智能体能够跨团队、产品和组织发现能力、通信和委派任务。A2A 技术指导委员会包括来自 AWS、Cisco、Google、IBM Research、Microsoft、Salesforce、SAP 和 ServiceNow 的代表。
